## 1. 背景与问题

### 1.1 业务背景
订单系统业务爆发式增长，日订单量达到 **100万+**，单库数据量突破 **1000万**，导致以下问题：

- ❌ 数据库写入压力大，TPS瓶颈明显
- ❌ 查询响应慢，慢SQL频发
- ❌ 单表数据量过大，索引效率下降
- ❌ 磁盘空间占用高，备份恢复时间长
- ❌ 数据库连接池耗尽，影响业务稳定性

### 1.2 分库分表目标
- ✅ 提升写入性能：支持日均100万+订单写入
- ✅ 优化查询性能：常用查询响应时间<100ms
- ✅ 提高系统可用性：分库分表后支持故障隔离
- ✅ 便于数据归档：历史数据独立管理
- ✅ 支持水平扩展：业务增长时可动态扩容

## 3. 方案设计

### 3.1 方案A：用户ID分库 + 订单ID分表

#### 3.1.1 分片策略

**分库策略**
- 分片键：`user_id`
- 分库数：4个库
- 分片算法：`user_id % 4`
- 库命名：`order_db_0`, `order_db_1`, `order_db_2`, `order_db_3`

**分表策略**
- 分片键：`id` (订单ID)
- 分表数：每个库16张表
- 分片算法：`id % 16`
- 表命名：`t_order_0`, `t_order_1`, ..., `t_order_15`

#### 3.1.2 数据分布

```
┌─────────────────────────────────────────────────────────────┐
│                     订单数据分布                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  user_id = 1, 5, 9, 13...  →  order_db_0                    │
│  user_id = 2, 6, 10, 14... →  order_db_1                    │
│  user_id = 3, 7, 11, 15... →  order_db_2                    │
│  user_id = 4, 8, 12, 16... →  order_db_3                    │
│                                                              │
│  每个库包含16张表 (t_order_0 ~ t_order_15)                   │
│  order_id = 1, 17, 33...    →  t_order_1                    │
│  order_id = 2, 18, 34...    →  t_order_2                    │
│  ...                                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 3.1.3 架构图

```
                              应用层
                                 │
                                 ▼
                         Sharding-JDBC
                                 │
         ┌───────────┬───────────┼───────────┬───────────┐
         ▼           ▼           ▼           ▼           ▼
    order_db_0  order_db_1  order_db_2  order_db_3  (可扩展)
         │           │           │           │
    ┌────┴────┐  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
    │ table 0 │  │ table 0 │  │ table 0 │  │ table 0 │
    │ table 1 │  │ table 1 │  │ table 1 │  │ table 1 │
    │   ...   │  │   ...   │  │   ...   │  │   ...   │
    │ table15 │  │ table15 │  │ table15 │  │ table15 │
    └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘
         │           │           │           │
    ┌────┴────┐  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
    │ item 0  │  │ item 0  │  │ item 0  │  │ item 0  │
    │ item 1  │  │ item 1  │  │ item 1  │  │ item 1  │
    │   ...   │  │   ...   │  │   ...   │  │   ...   │
    │ item15  │  │ item15  │  │ item15  │  │ item15  │
    └─────────┘  └─────────┘  └─────────┘  └─────────┘
```

#### 3.1.4 优点
✅ **查询性能优**：用户订单查询只需访问单一库
✅ **数据均衡**：用户ID取模保证数据均匀分布
✅ **扩展性强**：可动态增加库数，只需修改取模数
✅ **事务简单**：同一用户的数据在同一库，支持单库事务
✅ **热点分散**：避免单个用户订单过多导致的单表热点

#### 3.1.5 缺点
❌ **跨库查询**：按时间范围查询需要扫描所有库
❌ **订单号查询**：需要在订单号中编码用户ID
❌ **热点用户**：超级用户的订单可能集中在某张表
❌ **归档复杂**：历史数据归档需要跨库操作

### 3.2 方案B：时间分库 + 用户ID分表

#### 3.2.1 分片策略

**分库策略**
- 分片键：`create_time`
- 分库数：按月分库
- 分片算法：按月归档
- 库命名：`order_db_202401`, `order_db_202402`, ...

**分表策略**
- 分片键：`user_id`
- 分表数：每个库10张表
- 分片算法：`user_id % 10`
- 表命名：`t_order_0`, `t_order_1`, ..., `t_order_9`

#### 3.2.2 数据分布

```
┌─────────────────────────────────────────────────────────────┐
│                     订单数据分布                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  2024-01  →  order_db_202401                               │
│  2024-02  →  order_db_202402                               │
│  2024-03  →  order_db_202403                               │
│  ...                                                       │
│                                                              │
│  每个库包含10张表 (t_order_0 ~ t_order_9)                    │
│  user_id = 1, 11, 21...  →  t_order_1                      │
│  user_id = 2, 12, 22...  →  t_order_2                      │
│  ...                                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2.3 优点
✅ **时间查询优**：按时间范围查询只需访问特定月份的库
✅ **数据归档**：历史数据可以独立备份和归档
✅ **冷热分离**：可以针对不同时期的数据做不同的优化
✅ **报表分析**：支持按月/按年的统计分析

#### 3.2.4 缺点
❌ **用户查询差**：查询用户订单需要跨库合并
❌ **写入热点**：当前月份的库写入压力集中
❌ **事务复杂**：跨库事务处理困难
❌ **扩容受限**：需要提前规划创建下月库

## 4. 推荐方案详解

### 4.1 选择方案A的理由

| 评估维度 | 权重 | 方案A得分 | 方案B得分 | 说明 |
|---------|------|----------|----------|------|
| 用户订单查询 | 30% | 95分 | 40分 | 方案A单库查询，性能优异 |
| 写入性能 | 25% | 90分 | 70分 | 方案A均匀分布，方案B有热点 |
| 扩展性 | 20% | 95分 | 60分 | 方案A可动态扩容 |
| 实现复杂度 | 15% | 80分 | 60分 | 方案A实现更简单 |
| 维护成本 | 10% | 85分 | 65分 | 方案A维护成本低 |
| **总分** | 100% | **90分** | **60分** | **方案A胜出** |

**核心优势：**
1. 电商系统最核心的查询是"查询我的订单"，方案A完美匹配
2. 单库查询避免跨库合并，响应时间可控
3. 技术栈成熟，Sharding-JDBC配置简单
4. 后期扩展方便，可从4库扩展到8库、16库

#### 5.2.4 修改订单号生成策略

```java
/**
 * 订单号生成规则：
 * 格式：{分片标识}{用户ID补位}{时间戳}{随机数}
 * 示例：O000000123202401241530451234567
 * 
 * 分片标识：O (Order)
 * 用户ID：补位到7位（从订单号中提取分库键）
 * 时间戳：yyyyMMddHHmmss
 * 随机数：6位
 */
public class OrderNoGenerator {
    
    public static String generate(Long userId) {
        String prefix = "O";
        String userIdStr = String.format("%07d", userId);
        String timestamp = LocalDateTime.now()
            .format(DateTimeFormatter.ofPattern("yyyyMMddHHmmss"));
        String random = String.format("%06d", new Random().nextInt(1000000));
        
        return prefix + userIdStr + timestamp + random;
    }
    
    public static Long extractUserId(String orderNo) {
        if (orderNo == null || orderNo.length() < 9) {
            return null;
        }
        return Long.parseLong(orderNo.substring(1, 8));
    }
}
```

#### 5.2.5 实现双写逻辑

```java
@Service
public class OrderServiceImpl implements OrderService {
    
    @Autowired
    private OrderMapper orderMapper;  // 原有单库mapper
    
    @Autowired
    private ShardingOrderMapper shardingOrderMapper;  // 分库分表mapper
    
    /**
     * 双写：同时写入旧库和新库
     */
    @Override
    @Transactional
    public Order createOrder(Order order) {
        // 1. 生成分布式ID和订单号
        Long orderId = snowflakeIdGenerator.nextId();
        order.setId(orderId);
        order.setOrderNo(OrderNoGenerator.generate(order.getUserId()));
        
        // 2. 写入旧库（兼容现有业务）
        orderMapper.insert(order);
        
        // 3. 写入新库（分库分表）
        try {
            shardingOrderMapper.insert(order);
        } catch (Exception e) {
            // 记录日志，不影响主流程
            log.error("分库分表写入失败，orderId: {}", orderId, e);
        }
        
        return order;
    }
}
```

### 6.2 分片算法说明

**行表达式分片算法（Inline）**
```
ds${user_id % 4}
  └─ user_id = 1 → ds0
  └─ user_id = 2 → ds1
  └─ user_id = 3 → ds2
  └─ user_id = 4 → ds3

t_order_${id % 16}
  └─ id = 1 → t_order_1
  └─ id = 17 → t_order_1
  └─ id = 2 → t_order_2
  └─ id = 18 → t_order_2
```

### 7.3 数据一致性校验

```java
/**
 * 数据一致性校验
 */
@Component
public class DataConsistencyChecker {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private ShardingOrderMapper shardingOrderMapper;
    
    public void checkConsistency() {
        // 1. 对比总数
        int oldCount = orderMapper.selectCount(null);
        int newCount = shardingOrderMapper.selectCount(null);
        
        if (oldCount != newCount) {
            log.error("数据总数不一致: 旧库={}, 新库={}", oldCount, newCount);
        }
        
        // 2. 抽样对比
        List<Order> oldOrders = orderMapper.selectList(
            new Page<>(1, 100), null
        );
        
        for (Order oldOrder : oldOrders) {
            Order newOrder = shardingOrderMapper.selectById(oldOrder.getId());
            if (newOrder == null || !oldOrder.equals(newOrder)) {
                log.error("数据不一致: orderId={}", oldOrder.getId());
            }
        }
    }
}
```

## 9. 风险与注意事项

### 9.1 风险分析

| 事务问题 | 高 | 避免跨库事务、使用补偿机制 |

### 9.2 注意事项

#### 9.2.1 分片键选择

✅ **推荐：user_id**
- 查询订单最常用的条件
- 数据分布均匀
- 业务意义明确

❌ **不推荐：create_time**
- 无法定位到具体用户
- 热点数据集中
- 查询效率低

#### 9.2.2 分片数量

**分库数量：4个**
- 当前数据量：100万
- 预期增长：每月300万
- 单库容量：~750万（合理范围）

**分表数量：16个**
- 单表数据量：~60万
- 查询性能：优秀
- 扩容空间：充足

#### 9.2.3 订单号设计

**格式：O{用户ID补位}{时间戳}{随机数}**
```java
// 示例：O000000123202401241530451234567
//    └─ 固定前缀
//       └─ 用户ID（7位补零）
//          └─ 时间戳（14位）
//             └─ 随机数（6位）
```

**优势：**
- 可从订单号中提取user_id作为分片键
- 保证全局唯一性
- 包含时间信息，便于排查

#### 9.2.4 跨库查询处理

**方案1：使用Sharding-JDBC的hint路由**
```java
// 强制路由到指定库
HintManager hintManager = HintManager.getInstance();
try {
    hintManager.setDatabaseShardingValue(0);
    List<Order> orders = orderMapper.selectList(wrapper);
} finally {
    hintManager.close();
}
```

**方案2：数据聚合（应用层）**
```java
// 查询所有库，在应用层聚合
List<Order> allOrders = new ArrayList<>();
for (int i = 0; i < 4; i++) {
    HintManager hintManager = HintManager.getInstance();
    try {
        hintManager.setDatabaseShardingValue(i);
        List<Order> orders = orderMapper.selectList(wrapper);
        allOrders.addAll(orders);
    } finally {
        hintManager.close();
    }
}
```

#### 9.2.5 事务处理

**原则：避免跨库事务**

❌ **错误做法：**
```java
@Transactional
public void transferOrder(Long fromUserId, Long toUserId, Long orderId) {
    // 跨库操作，事务失效
    orderMapper.updateStatus(orderId, "CANCELLED");
    orderMapper.createOrder(toUserId, ...);
}
```

✅ **正确做法：**
```java
// 使用补偿机制
public void transferOrder(Long fromUserId, Long toUserId, Long orderId) {
    // 1. 取消原订单
    orderMapper.updateStatus(orderId, "CANCELLED");
    
    // 2. 创建新订单
    try {
        orderMapper.createOrder(toUserId, ...);
    } catch (Exception e) {
        // 3. 补偿：恢复原订单状态
        orderMapper.updateStatus(orderId, "CREATED");
        throw new RuntimeException("订单转移失败", e);
    }
}
```

### 10.2 常见问题

A: 避免跨库JOIN。可以在应用层分两次查询，或者使用数据冗余。

A: 优先选择查询条件中最常用的字段，且数据分布均匀。
