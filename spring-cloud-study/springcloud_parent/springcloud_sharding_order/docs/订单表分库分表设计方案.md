## 一、项目背景

### 1.1 业务背景

订单服务是核心业务系统，承载着全平台的订单创建、查询、更新等功能。随着业务快速发展，订单数据量持续增长，目前已达到 **10亿条数据**，单表存储和查询性能成为系统瓶颈。

### 1.2 问题分析

| 问题 | 描述 | 影响 |
|------|------|------|
| 单表数据量过大 | t_order和t_order_item单表数据超过10亿 | 查询性能下降，索引失效 |
| 写入性能瓶颈 | 高峰期写入延迟高 | 订单创建成功率下降 |
| 存储成本高 | 单表占用存储空间大 | 数据库维护成本高 |
| 扩展性差 | 单表无法水平扩展 | 无法应对未来增长 |

### 1.3 改造目标

- ✅ **性能提升**：订单查询响应时间降低至P99 < 200ms
- ✅ **水平扩展**：支持按月分表，未来可按需扩展
- ✅ **数据迁移**：实现零停机迁移，业务无感知
- ✅ **平滑过渡**：支持灰度切换，风险可控

## 二、总体设计

### 2.1 设计原则

1. **最小侵入**：使用AOP实现双写，业务代码改动最小
2. **可回滚**：每个阶段都可快速回退，确保系统稳定
3. **灰度发布**：小步快跑，逐步切换流量
4. **数据一致性**：保证迁移期间数据一致性
5. **可监控**：全程监控迁移进度和业务指标

### 2.2 技术选型

| 组件 | 版本 | 说明 |
|------|------|------|
| ShardingSphere-JDBC | 5.2.1 | 分库分表中间件 |
| Spring AOP | 2.x | 实现双写逻辑 |
| MyBatis | 2.2.0 | ORM框架 |
| Druid | 1.2.8 | 数据库连接池 |
| Apollo | - | 配置中心（可选） |

### 2.3 分片策略

#### 分片键选择
- **订单表（t_order）**：`order_no`（订单编号）
- **订单明细表（t_order_item）**：`order_no`（订单编号）

#### 分片算法
- **算法类型**：基于雪花算法ID的时间戳按月分片
- **分片规则**：从 `order_no` 中提取年月，路由到对应月表
- **表命名规范**：`t_order_YYYYMM`、`t_order_item_YYYYMM`

#### 分片范围
- **起始月份**：202510
- **结束月份**：202610
- **总表数**：13个月 × 2张表 = 26张表

#### 绑定表配置
```
t_order 和 t_order_item 配置为绑定表
- 确保同一订单的订单和明细在同一个分片
- 避免跨库关联查询
- 提升查询性能
```

### 2.4 迁移策略

采用 **双写 + 数据迁移 + 灰度切换** 的零停机迁移策略：

```
┌─────────────────────────────────────────────────────────────┐
│                     零停机迁移整体流程                      │
└─────────────────────────────────────────────────────────────┘

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ 阶段一：     │ →   │ 阶段二：     │ →   │ 阶段三：     │
│ 离线迁移     │     │ 启用双写     │     │ 灰度切读     │
└──────────────┘     └──────────────┘     └──────────────┘
        │                    │                    │
        ▼                    ▼                    ▼
   搬迁历史数据        同时写入新老表         逐步切换读流量
   (202510-当前月)    观察新表稳定性         5%→20%→50%→100%
        │                    │                    │
        ▼                    ▼                    ▼
   数据校验              监控双写成功率          全量切换
   保证一致性            失败率<0.1%             关闭双写
                                                      │
                                                      ▼
                                              ┌──────────────┐
                                              │ 阶段四：     │
                                              │ 清理收尾     │
                                              └──────────────┘
                                                      │
                                                      ▼
                                              下线老表数据源
```

**核心组件设计**：

##### 3.1 分片算法（OrderMonthShardingAlgorithm）

```
功能描述：
  从order_no中提取年月，路由到对应分片表

输入：
  order_no（雪花算法生成的订单号）

输出：
  表名（如：t_order_202510）

算法逻辑：
  1. 从order_no中提取时间戳（右移22位 + 起始时间戳）
  2. 将时间戳转换为LocalDate
  3. 格式化为年月字符串（yyyyMM）
  4. 返回表名 t_order_yyyyMM
```

##### 3.2 双写切面（DualWriteAspect）

```
功能描述：
  拦截Mapper的写操作，根据配置决定写哪些数据源

拦截目标：
  - com.zf.mapper.*.insert(..)
  - com.zf.mapper.*.update(..)
  - com.zf.mapper.*.delete(..)

核心逻辑：
  1. 读取配置 migration.write-source
  2. 根据配置执行写操作：
     - old: 只写老表（原有逻辑）
     - new: 只写新表（ShardingSphere数据源）
     - both: 先写老表，再写新表
  3. 双写时，新表写入失败记录日志但不影响主流程
  4. 使用ThreadLocal避免AOP嵌套

配置示例：
  migration.write-source: old  → 只写老表
  migration.write-source: new  → 只写新表
  migration.write-source: both → 同时写新老表
```

##### 3.3 数据迁移服务（DataMigrationService）

```
功能描述：
  按月分批迁移历史数据，支持断点续传

核心方法：
  - createMigrationTask(): 创建迁移任务
  - executeMigration(taskId): 执行迁移任务
  - migrateByMonth(table, month): 按月迁移
  - migrateBatch(table, month, offset, limit): 分批迁移
  - validateData(table, month): 数据校验
  - rollbackMigration(taskId): 回滚迁移

迁移流程：
  1. 创建迁移任务记录
  2. 查询老表该月数据总量
  3. 分页查询老表数据（每批batch_size条）
  4. 批量写入新表（使用ShardingSphere）
  5. 记录迁移记录（幂等性）
  6. 更新迁移进度
  7. 全部完成后更新任务状态

容错机制：
  - 失败自动重试（最多3次）
  - 支持断点续传
  - 迁移记录幂等性保证
```

#### R001：双写失败导致数据不一致

- 定期对账新老表数据
- 使用消息队列保证最终一致性

**应急措施**：
- 立即切换到只写老表
- 补写新表缺失数据
- 分析失败原因并修复

- 限制迁移并发数（如4个线程）

- 暂停迁移任务
- 调整迁移参数
- 延长迁移窗口期
